<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| renderer -> preload/main (IPC) | Renderer runs web content (`contextIsolation:true`, `sandbox:true`); every native capability crosses `window.nyanko` -> `ipcMain.handle`. Renderer-supplied arguments are untrusted here. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-01 | Information Disclosure / EoP | `readAppDataFile` IPC handler | high | mitigate | Keep the existing hard whitelist `READABLE_APPDATA_FILES = {port, instance_token}`; do NOT widen to arbitrary paths. Reject any other name (returns null). |
| T-03-02 | Tampering / EoP | `openExternal` / `openPath` / `revealItemInDir` handlers | high | mitigate | `openExternal` accepts only `^https?://` strings (no `file:`, no custom protocol handlers, no exe launch). `openPath`/`revealItemInDir` reject any string containing `://` (local library paths only). |
| T-03-03 | Elevation of Privilege | preload contextBridge surface | high | mitigate | Expose only specific typed methods on `window.nyanko`; never expose raw `ipcRenderer` or a generic `invoke`. |
| T-03-04 | Spoofing | `notify` handler | low | accept | Title/body are the app's own i18n strings; notification spoofing risk is negligible and not worth constraining. |
| T-03-SC | Tampering | npm installs | n/a | n/a | This phase adds ZERO new dependencies (notifications/dialog/shell are Electron built-ins; Discord/updater are Phase 4/5 stubs). No package legitimacy gate required. |
</threat_model>
