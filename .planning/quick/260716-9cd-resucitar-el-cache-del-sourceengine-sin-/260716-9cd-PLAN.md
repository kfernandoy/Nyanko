## 1. El criterio de WR-03 ya existe: no inventes otro

`errors.py:34` ya tiene la distincion, y es la unica que hay:

| Error | `source_error_action()` | ¿Cache? |
|---|---|---|
| `SourceRateLimitError` | `"esperar"` | **NO** — la fuente pide que pares |
| `SourceNetworkError` | `"reintentar"` | si |
| `SourceParseError` / `NotFound` / `Unsupported` | `"actualizar_la_fuente"` | si |

El criterio es: **`"esperar"` no se sirve de cache; todo lo demas si.** No `isinstance(error,
SourceRateLimitError)` — eso duplicaria una decision que `source_error_action()` ya posee, y si mañana
otro error mapea a `"esperar"`, la version con `isinstance` no se entera y la del helper si.

Ojo con la tentacion de apretar mas: `SourceParseError` mapea a `"actualizar_la_fuente"` y **debe
seguir sirviendo cache** — `test_chapters_returns_good_cache_after_source_parse_error` lo exige. El
criterio NO es "solo reintentar".

Observacion, **no la arregles aqui**: un 404 con cache caliente y mismo registry serviria cache rancio.
Hoy pasa igual y esta fuera del alcance de WR-03 (que es rate limit). Anotado, no tocado.

## 2. CR-03: derivar al leer, no mantener el invariante en 3 sitios

`build_source_registry` se llama en **tres** sitios (`main.py:1469` lifespan, `2130` alta, `2146` baja).
La solucion obvia — un helper que ponga registry + engine en los tres — mantiene un invariante en tres
escritores. Un cuarto sitio que llame a `build_source_registry` a pelo reintroduce CR-03 en silencio.

**Este proyecto ya decidió esto, y esta escrito en STATE.md (Decisions, Phase 01):**

> «progress tiene cuatro escritores que no lo tocaran: un invariante mantenido en cuatro sitios se
> rompe, uno derivado al leer (`progress.effective_chapter`) no»

Mismo patron aqui. El engine se **deriva de la identidad del registry al leer**:

```python
def _source_engine(request: Request) -> SourceEngine:
    registro: SourceRegistry = request.app.state.source_registry
    engine: SourceEngine | None = getattr(request.app.state, "source_engine", None)
    # El engine vive lo que viva el registry que envuelve: si el registry se reconstruye
    # (alta/baja de carpeta), cambian las raices y el cache DEBE morir con el.
    if engine is None or engine.registry is not registro:
        engine = SourceEngine(registro)
        request.app.state.source_engine = engine
    return engine
```

Propiedades que salen gratis y por las que se elige esta forma sobre el helper:

- **Los 3 sitios de rebuild no se tocan.** Cero riesgo de romper `test_lifespan_builds_source_registry_once`
  (afirma `len(calls) == 1`): este cambio no añade ni una llamada a `build_source_registry`.
- **Un rebuild tira el cache por construccion**, sin que nadie tenga que acordarse.
- **WR-06 no empeora**: el engine no puede quedarse envuelto en un registry viejo.
- Sin `await` entre el check y el set (funcion sincrona) → atomico en el event loop. **No hace falta lock.**

**La trampa que hay que evitar:** memoizar con `if not hasattr(app.state, "source_engine")` a secas.
Eso deja el engine envuelto en el registry VIEJO para siempre → una carpeta añadida en caliente se
vuelve invisible → **empeora WR-06** y rompe el test del punto 3.

Requiere exponer el registry en `SourceEngine` (3 lineas), en vez de que `main.py` meta mano en
`engine._registry`:

```python
    @property
    def registry(self) -> SourceRegistry:
        return self._registry
```

## 3. Un test que YA EXISTE guarda las dos mitades de esto (gratis)

`test_manga_api.py:269 test_alta_y_baja_de_carpeta_refrescan_el_registry_sin_reiniciar`:
alta → GET `visible` **200** → baja → GET `invisible` **404**.

Ese test castiga las dos formas de equivocarse:

- Memo naif (`hasattr`) → tras el alta el engine sigue envuelto en el registry vacio → `visible` da 404. **Rojo.**
- Engine persistente **sin** tirar el cache en el rebuild → tras la baja el cache tiene la clave, la
  fuente lanza `SourceNotFoundError` (`"actualizar_la_fuente"` → hay fallback) y el fallback devuelve
  cache → `invisible` da **200** en vez de 404. **Rojo.**

Con la identidad del registry, ambos salen verdes. **No lo toques.** Si se pone rojo, el arreglo esta mal.

## 4. Los helpers de test NO hay que tocarlos (verificado con grep)

`_cliente()` (`test_manga_api.py:59`) y `_installed_registry()` (`test_source_api.py`) instalan **solo**
`app.state.source_registry` y nunca un engine. Un `_source_engine()` que hiciera

`source_engine` filtrado de otro test se quedaria pegado. El check de identidad lo neutraliza (registry
distinto → engine nuevo). **Sin contaminacion entre tests.**
